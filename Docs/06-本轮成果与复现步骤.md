## 数据来源说明（重要，可能需要你决策）

Kaggle官方ASAP-AES数据集需要账号登录/接受比赛协议，这一步没法自动化，本轮没有你的Kaggle账号。改用了HuggingFace上"llm-aes"组织发布的非gated镜像（来源于开源项目[Xiaochr/LLM-AES](https://github.com/Xiaochr/LLM-AES)），数据内容和Kaggle官方一致（同样的essay文本、同样的domain1_score），只是分发渠道不同。

**如果你更倾向用Kaggle官方渠道**（比如担心答辩时被问"这个数据集从哪来的，为什么不是官方链接"），操作很简单：去Kaggle下载`training_set_rel3.tsv`放到`data/raw/`下，重新跑一遍`python -m src.data_pipeline.clean`，代码会自动优先使用Kaggle文件而不是HF镜像（数据内容应该完全一致，跑出来的清洗后条数、QWK应该也很接近）。这个选择目前是我按"没有Kaggle账号、需要能自动化"的约束做的判断，不是唯一解，`TODO.md`里也记了这条。

```bash

source .venv/bin/activate

```

应该会看到：真实的DeepSeek生成反馈（中文，针对具体作文内容）、一个0-1之间的评分、SQLite历史记录写入成功、过短作文被正确拒绝。如果想在浏览器里点着看：

```bash
streamlit run app.py --server.port 8502 --server.address 0.0.0.0
```
（用一个新端口8502，不要用8501，那个端口是RetinaScope生产服务在用——见下面"需要你知道的一件事"）然后在你自己的电脑浏览器访问`http://<服务器IP>:8502`（需要服务器防火墙/安全组放通这个端口，或者用SSH端口转发`ssh -L 8502:localhost:8502 retinascope-server`更安全）。

### 2. 需要你知道的一件事（已经在对话里跟你说过，这里存档）

本轮操作过程中，我执行`pkill -f 'streamlit run app.py'`想停掉自己起的一个测试进程，结果这个模式匹配同时命中了这台服务器上已经在跑的、你的RetinaScope生产Docker容器（`retinascope_app`），导致它被杀掉、中断了大约1分钟。Docker的自动重启策略(`unless-stopped`)在几十秒内把它拉起来了，`docker ps`确认现在是健康状态，我也用`docker inspect`核对了重启时间点确实就是那次误操作导致的。你当时已经确认"继续，以后只用精确PID"，这件事就到此为止，不需要你再做什么，只是存档一下避免你从别处发现时觉得奇怪。以后我在这台机器上停任何进程都只用精确PID，不会再用模式匹配。
