# 08 部署操作手册

> 负责人：毛陈荣。负责范围：把系统部署到自有服务器`121.41.238.92`，让答辩现场能通过浏览器访问。

## 一、部署目标服务器情况（部署前必须知道，不是空机器）

- 别名`deploy-server`，配置在`~/.ssh/config`（`root`用户，22端口，`IdentityFile ~/.ssh/id_rsa`，已测试连通）。**这台机器和用于训练模型的`retinascope-server`是两台不同的机器**，不要混淆，训练不用碰这台。
- **磁盘部署前9.3G可用**（根分区40G，已用28G），装完全部Python依赖后剩约6.2G——够用但不宽裕，不要在这台机器上跑训练/存大量数据/装不必要的重依赖。
- **内存只有1.6G，这是比磁盘更容易踩坑的限制**：`EssayScorer`加载一次微调DistilBERT模型就要占约800MB常驻内存。Streamlit常驻服务本身已经加载了一份模型，**如果这时候再另起一个进程跑`scripts/e2e_graph_test.py`之类的验证脚本，会同时加载第二份模型，直接触发OOM killer杀进程**（本轮部署时真实发生过，`dmesg`里能看到`Out of memory: Killed process ... (python)`）。内存被打满时`sshd`本身也会响应迟缓，表现成"SSH连接一直banner exchange超时"，容易误判成网络问题。**要在这台机器上跑任何加载模型的验证脚本，必须先用精确PID停掉正在跑的Streamlit，跑完再重新启动**，不要让两个模型加载进程同时存在。
- **端口`80`/`8080`/`8501`/`8502`都已经被占用**：`8501`是一个叫`ophthalmic-ai`的Docker容器（`docker ps`显示"Up 2 months (healthy)"，是田溯开另一个已上线的医疗项目），**绝对不能碰这个容器**（更不能用`pkill -f`之类的模式匹配去停任何进程，本项目已经在训练服务器上因为这个方式误杀过一次生产容器，教训写在`CLAUDE.md`）。**我们的应用固定用端口`8503`**（已确认当前空闲，部署前务必再用`ss -tlnp`确认一遍没被占用）。
- **云安全组默认没有放通8503**：即使服务器本机`curl localhost:8503`返回200，外网也访问不了，这一步跟服务器本机的`iptables`/`ufw`无关（本轮实测两者都没有拦截），是阿里云ECS控制台的安全组规则没放行，**必须登录阿里云控制台手动加一条TCP 8503入站规则**（实例`i-bp13v9vqfgbj2iygazp9`，区域`cn-hangzhou`）。这一步SSH改不了、也没有装云厂商CLI工具能代劳，只能去控制台点。**本轮已经放通并验证过**（`curl http://121.41.238.92:8503`外网返回200），以后如果服务器重建/安全组规则被重置，记得这一步。
- Docker本身已经装好（29.4.1），但没有现成的nginx。**当前团队决策是先用最简单的venv+nohup方式部署**（见下方"标准部署流程"），不是完整的Docker容器化——4天工期下先求"能跑、能访问"，Docker化是锦上添花，如果时间充裕可以按第三节的思路补。

## 二、标准部署流程（**已实际执行过一次**，代码在`/root/sukai/`，不是早期草稿里写的`/root/huibi/`）

```bash
# 1. 把代码同步到服务器（本地开发机没有rsync，用tar+scp；注意不能用--exclude-from=.gitignore
#    整个排除，那样会把models/下的真实权重文件也排除掉，部署出来的应用会降级成占位评分）
tar --exclude='.git' --exclude='.venv' --exclude='__pycache__' --exclude='.env' -czf huibi_deploy.tar.gz .
scp huibi_deploy.tar.gz deploy-server:/root/huibi_deploy.tar.gz
ssh deploy-server "mkdir -p /root/sukai && cd /root/sukai && tar xzf /root/huibi_deploy.tar.gz && rm /root/huibi_deploy.tar.gz"

# 2. 登录服务器，装依赖（无GPU，torch要装CPU-only版，否则pip默认拉CUDA版体积大很多）
ssh deploy-server
cd /root/sukai && python3 -m venv .venv && source .venv/bin/activate
pip install --no-cache-dir torch --index-url https://download.pytorch.org/whl/cpu
pip install --no-cache-dir -r requirements.txt

# 3. 服务器上单独准备.env（不要用tar/rsync同步真实Key，用scp单独传一次）
scp .env deploy-server:/root/sukai/.env

# 4. 构建RAG知识库（chroma_kb被.gitignore排除，没有跟着代码一起传过去，要在服务器上现场生成）
ssh deploy-server "cd /root/sukai && source .venv/bin/activate && export HF_ENDPOINT=https://hf-mirror.com HF_HUB_DISABLE_XET=1 && python -m src.rag.build_kb"

# 5. 启动/停止用脚本，不要手写nohup命令（脚本内部只用PID文件记录的精确PID管理进程）
powershell -File scripts/deploy_start.ps1   # 本地跑，启动+自动验证HTTP 200
powershell -File scripts/deploy_stop.ps1    # 本地跑，停止

# 6. 确认防火墙/安全组放通8503端口
```

**部署前必须做的检查**（不是可选步骤）：

```bash
docker ps                                  # 确认没有误碰到ophthalmic-ai容器
ss -tlnp | grep 8503                       # 确认8503确实空闲
df -h /                                    # 确认磁盘还有空间装依赖（pip install torch+transformers体积不小）
```

**部署后必须做的验证**（**已完成，外网实测通过**）：

```bash
curl -s -o /dev/null -w '%{http_code}\n' http://121.41.238.92:8503   # 已验证返回200
```

**仍然建议**从答辩现场实际用的网络（校园网/手机热点等）再单独访问一次`http://121.41.238.92:8503`——本轮验证是在开发者所在网络下做的，答辩现场的校园网/内网出口策略可能不一样，不能完全替代现场实测，提前确认一次比较稳妥。

**日常启停用这两个脚本**（`scripts/deploy_start.ps1`/`scripts/deploy_stop.ps1`，本地PowerShell跑，通过SSH远程操作）：
- 启动脚本会先检查PID文件确认没有重复启动，启动后自动`curl`验证HTTP 200。
- 停止脚本**只kill PID文件里记录的精确PID**，不用`pkill -f`模式匹配——这台服务器是共用机器，模式匹配可能误杀别人的进程（本项目在训练服务器上已经犯过一次这个错误，见`CLAUDE.md`"不要重新踩的坑"），部署服务器上必须更小心。

## 三、如果要做Docker容器化（可选，不是必须）

当前没有现成的`Dockerfile`。如果决定要做，思路：
1. 写`Dockerfile`：基于`python:3.10-slim`，`COPY requirements.txt`+`pip install`，`COPY`代码，`CMD ["streamlit", "run", "app.py", "--server.port=8503", "--server.address=0.0.0.0"]`。
2. 模型权重（`models/essay-scorer-*/v1/pytorch_model.bin`共约270MB）不进镜像，运行时用`scripts/download_models.py`从GitHub Release拉取，或者用`docker run -v`挂载宿主机已经下载好的目录，避免镜像体积失控（磁盘只剩9.3G，装不下太大的镜像）。
3. `.env`同样不要打进镜像，用`docker run --env-file .env`或`-v`挂载。
4. 跑起来后用`-p 8503:8503`映射端口，不要用`8501`（已被`ophthalmic-ai`占用，容器化不代表端口冲突问题会自动消失）。

**这条不是当前的主线方案**，只是留一个可执行的思路；如果时间不够，直接用第二节的venv+nohup方案完成部署即可，不要为了"看起来更专业"而在Day4临近截止时才折腾容器化，参考`CLAUDE.md`"优先级原则"——主链路能跑通、能演示比部署方式本身更重要。

## 四、离线演示预案（部署之外必须准备的兜底）

万一答辩现场网络连不上`121.41.238.92:8503`（校园网限制、服务器故障、现场没有可用网络等），必须有兜底：
1. 提前用几个不同题材/水平的作文跑一遍完整流程，截图或录屏保存下来。
2. 准备好本地能起服务的方案（比如答辩用的笔记本本地`streamlit run app.py`，前提是本地环境能跑起完整依赖）。
3. 最保底：录制一份完整的演示视频（Day4 checklist里"全员"这条任务），万一现场所有方案都失败，播放视频保证不会"什么都展示不出来"。

## 五、可能被追问的问题

- **"为什么不整体上云/用Docker/K8s？"** —— 4天工期，团队已有的自有服务器满足答辩演示需求，venv+nohup最简单可控；Docker容器化的思路已经想清楚（见第三节），但优先级低于主链路稳定性。
- **"这台服务器还跑着别的服务，会不会互相影响？"** —— 检查过端口冲突（`8501`被占用），我们的应用用独立端口`8503`，不共享同一个Python环境（各自`venv`），进程级别不冲突；但这台机器只有1.6G内存，**内存是真实会互相影响的资源**——本轮部署时验证脚本和Streamlit同时加载模型就触发过OOM，所以任何需要加载模型的操作都要避开Streamlit正在跑的时间窗口，见上方"内存只有1.6G"那条。
- **"如果现场访问不了怎么办？"** —— 见第四节离线预案，不是临时想的，是提前准备好的。
