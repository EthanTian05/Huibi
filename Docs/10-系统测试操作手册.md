# 10 系统测试操作手册

> 负责人：史翼嘉。负责范围：所有测试/验证脚本（`scripts/`目录），端到端联调，记录并推动修复发现的问题。

## 一、负责范围里有哪些文件

| 文件 | 验证什么 | 是否需要完整依赖 |
|---|---|---|
| `scripts/check_llm_key.py` | `.env`里的DeepSeek/GLM Key能不能真的调通 | 不需要，纯标准库urllib |
| `scripts/smoke_test_nodes.py` | intake校验、grammar_check_node规则库、SQLite读写往返 | 不需要（不测scoring_tool_node的真实模型路径） |
| `scripts/e2e_graph_test.py` | 完整LangGraph链路（含真实DeepSeek调用+真实评分+真实RAG检索）+ 5个边界案例 | 需要（torch/transformers/langgraph/sklearn） |
| `scripts/eval_zero_shot_llm.py` | DeepSeek零样本打分 vs 训练模型的QWK对比 | 部分需要（pandas+urllib，不需要torch） |
| `scripts/eval_same_sample_and_diagnose_set8.py` | 同批次公平对比 + essay_set 8误差诊断 | 需要（torch/transformers/sklearn） |

## 二、操作手册（按依赖复杂度从低到高）

### 1. 不需要装任何依赖就能跑的（本地开发机直接用）

```bash
python scripts/check_llm_key.py
PYTHONPATH=. python scripts/smoke_test_nodes.py
```

`smoke_test_nodes.py`目前有4个测试函数：`test_intake_validator`（过短/正常长度两种输入）、`test_scoring_tool_stub`（**注意**：如果`models/essay-scorer-*/v1`权重文件已经存在本地，这个测试会尝试走真实推理路径，需要transformers/sklearn，本地开发机没装的话这一条会报`ModuleNotFoundError`，是环境限制不是代码问题，需要在完整依赖的机器上验证）、`test_grammar_check_rules`（验证规则库真实检测出拼写/重复/双重否定/情态动词误用/小写i这几类问题）、`test_db_roundtrip`（SQLite两次写入+历史查询）。

### 2. 需要完整依赖的（训练服务器或部署服务器上跑）

```bash
PYTHONPATH=. python scripts/e2e_graph_test.py
```

跑之前记得设离线HF缓存环境变量（否则可能因为网络问题在加载模型这一步失败，见`09`号文档第二节第3条）：

```bash
export HF_HOME=<缓存目录>
export HF_HUB_OFFLINE=1
export HF_HUB_DISABLE_XET=1
```

这个脚本会依次跑：
1. 一篇正常长度作文，走完整链路，断言反馈/建议非空、SQLite正确写入。
2. 一篇过短作文，断言被正确短路拒绝、不调用LLM。
3. **Day3新增的5个边界案例**（`test_edge_cases`函数）：
   - 正好20词（`MIN_WORDS`边界，应该通过校验）
   - 正好1000词（`MAX_WORDS`边界，应该通过校验）
   - 1001词（超过`MAX_WORDS`一词，应该被拒绝）
   - 语法错误密集的作文（验证`grammar_check_node`不会因为错误多而崩溃，且确实拉低了language分项）
   - essay_set 8的作文（分值范围最宽0-60，验证跨essay_set的分数归一化/反归一化不出错）

**已在训练服务器上实测全部通过，无异常抛出**（`Docs/Progress.md`第八轮记录）。

### 3. 零样本对比 + essay_set 8诊断（可复现，但会消耗真实API调用/需要GPU）

```bash
# 第一步：抽样80条（8个essay_set各10条，种子42）跑DeepSeek零样本打分
python scripts/eval_zero_shot_llm.py --n-per-set 10 --test-csv <test.csv路径>

# 第二步：同样80条essay_id，跑真实训练模型（默认使用A:0.95、B:0.05非等权融合）+ essay_set 8全量诊断（需要在有GPU/完整依赖的机器上跑）
python scripts/eval_same_sample_and_diagnose_set8.py --sample-ids-json <上一步保存的essay_id列表>
```

结果分别存在`models/zero_shot_llm_eval.json`、`models/same_sample_trained_eval.json`、`models/set8_diagnosis.json`；非等权权重的验证集搜索与最终结果见`models/weighted_ensemble_eval.json`，具体数字和结论见`09`号文档第四节。**重新跑一遍会消耗80次真实DeepSeek调用**，不是免费操作，不要没事就重跑，除非模型/规则库有实质性改动需要重新验证。

## 四、测试覆盖诚实说明（答辩被问"测试做得够不够"时怎么回答）

**已覆盖**：功能性边界测试（词数边界、跨essay_set、语法错误密集度、正常/异常输入）、端到端链路测试（含真实LLM/真实模型/真实RAG，不是mock）、数据往返一致性测试（SQLite）。

**没有覆盖、如实说明的部分**：
- 没做压力测试/并发测试（比如多用户同时提交），4人4天工期下这不是优先级，Streamlit本身单机部署也不是为高并发设计的。
- 没做对抗性输入测试（比如超长重复字符、特殊编码注入），`intake_validator_node`只做了词数校验，没有做内容安全过滤。
- LLM调用失败的重试策略目前依赖LangChain的`.with_fallbacks()`做供应商级别fallback（DeepSeek失败切GLM），但没有测试"两个供应商都失败"这种场景下前端具体会怎么报错。

如果评委追问这几点，如实说"这些是诚实列出的、4天工期下没时间做的部分，不是没考虑到"，比硬说"都测过了"更专业可信。

## 五、可能被追问的问题

- **"这套系统稳定吗，会不会随便输入就崩溃？"** —— 见第三节手动测试清单+第二节的边界案例自动化测试，覆盖了词数边界/语法密集/跨essay_set这几类容易出问题的场景，训练服务器上实测全部通过无异常。
- **"怎么知道评分模型不是运气好？"** —— 不是靠感觉，`training_log.json`里有完整的每轮epoch曲线（不是只有一个最终数字），测试集QWK是在完全没参与训练的1288+条数据上算出来的，且做过essay_set 8这种"模型明显较弱的场景"的专门诊断（见`09`号文档），敢暴露弱点比只讲亮点更可信。
- **"如果我现场随便写一篇作文提交会发生什么？"** —— 会走完整真实链路（不是演示模式/预设结果），评分、语法检查、反馈都是当场真实生成的，唯一的"非真实"是分项雷达图（启发式而非多头模型，已在`07`/`09`号文档说明），这点要主动讲清楚不要等被问到。
