# 03 模型训练与微调方案

## 模型定位：三条模型路径分工（已确认，见`TODO.md`本轮决策）

要求文档第6条"模型构建"允许"预训练模型-微调 / 调用通用大模型 / 自定义构建模型（加分项）"三选一，本项目**三条全做**，职责明确分开，同时把基础要求和加分项都做扎实，避免评委质疑"是不是只是套壳调了个API":

| 路径 | 用途 | 对应要求文档条目 |
|---|---|---|
| 通用大模型API（主力DeepSeek V4 Flash，兜底GLM-4.7-Flash，Key见本地`.env`） | 负责生成类任务：定性反馈撰写、RAG问答、个性化辅导建议 | 满足基础要求（"调用通用大模型"） |
| 自训练**微调**模型（评分器·路径A） | 负责判别类任务：给作文打量化分，基于预训练权重微调 | 满足"预训练模型-微调" |
| 自训练**自定义构建**模型（评分器·路径B，不依赖任何预训练权重） | 同样负责判别类任务，作为路径A的横向对比/融合对象 | 满足加分项"自定义构建模型" |

理由：量化评分需要和真实人工评分对齐、可用统一指标复现评估（QWK，见下），这件事大模型API直接"打分"做不到稳定校准；而定性反馈这种开放式生成任务恰恰是大模型API的强项。两个自训练评分模型（微调 vs 自建）能同时覆盖要求文档里"预训练模型-微调"和"自定义构建模型"两个选项，且两者可以做训练效果的横向对比，是很自然的答辩素材，比只做一条路径更扎实。

## 模型选型

### 路径A：微调（基于预训练权重）

- 基座：`bert-base-uncased` 或 `distilbert-base-uncased`（后者参数更小、训练更快，4天工期若GPU资源紧张优先选它）。
- 任务头：在`[CLS]`向量后接线性回归/多头回归层，输出归一化到[0,1]的分数（训练时按`02-数据集与数据处理方案.md`中的分essay_set归一化方式处理，推理后反归一化回真实分值区间）。

### 路径B：自定义构建（不依赖预训练权重，从零训练）

- 架构：Embedding层（随机初始化或加载开源静态词向量如GloVe，但不使用任何预训练Transformer权重）+ BiLSTM + Attention池化 + 回归/多头输出层，整体参数从随机初始化开始端到端训练。这是AES领域Transformer之前的经典架构思路，团队自行实现，不依赖HuggingFace预训练checkpoint，明确满足"自定义构建模型"的加分要求。
- 相比路径A的优势：训练更快（模型小、不依赖大型预训练权重下载），可以在CPU或入门级GPU上跑完整流程，作为路径A的效率/可解释性对照组，也降低"万一GPU资源紧张"的风险——两条路径可以并行推进，不互相卡进度。
- 两条路径最终可以简单加权融合（如平均）作为`EssayScorer`的默认输出，也可以在答辩时分别展示两者的QWK做对比，突出"团队既会用预训练微调，也有能力从零搭模型"。

### 通用设计（两条路径共用）

- **多头输出（已确认为主线方案，不再是条件项）**：两条路径的回归头都设计为多头输出——内容/结构/语言（如`essay_set` 7、8自带trait标注可直接监督训练；其余`essay_set`没有trait标注的，可用规则/启发式特征结合总分做弱监督分解，或者只在有trait标注的子集上训练多头模型，其余子集只用整体分），作为前端雷达图的分项数据来源。

## 训练配置

| 项 | 设置 |
|---|---|
| 损失函数 | MSE（回归）或Ordinal/序数分类损失（如把分数离散化为若干等级） |
| 评估指标 | **QWK（Quadratic Weighted Kappa，二次加权卡帕系数）** —— 与Kaggle ASAP竞赛官方评估指标一致，答辩时可以明确对标业界标准指标，体现方法的规范性 |
| 优化器 | AdamW |
| 学习率 | 2e-5 ~ 5e-5 区间网格试验 |
| Batch Size | 视GPU显存，8~16 |
| Epoch | 3~5轮，配合验证集早停 |
| 数据划分 | 按essay_set分层抽样，训练/验证/测试 |

## "225原则"（已确认）

要求文档第7条写"模型训练(225原则)"，本轮已确认含义，指的是**训练循环的标准结构**："2层循环 + 2个遍历对象 + 5个核心步骤"：

- **2层循环 / 2个遍历对象**：外层循环遍历训练轮次（epoch），内层循环遍历该轮次下的数据批次（batch）；
- **5个核心步骤**：每个batch内依次执行 **梯度清零 → 正向传播 → 损失计算 → 反向传播 → 参数更新**（`optimizer.zero_grad() → outputs = model(batch) → loss = criterion(outputs, labels) → loss.backward() → optimizer.step()`）。

这是训练循环最基础的写法，两条模型路径（微调/自定义构建）的训练脚本都要清晰体现这个结构，便于答辩时对着代码逐行讲清楚。**"225"是底线要求，不是上限**——团队可以在这个基础循环之上叠加更好的工程实践，包括但不限于：学习率调度（warmup+衰减）、梯度裁剪、混合精度训练、验证集早停（patience=2）、按`essay_set`分层的训练/验证/测试划分（8:1:1）。这些增强项写清楚"在225的基础上额外做了什么、为什么"，本身就是答辩时能加分的技术细节。

评估指标仍用QWK（见上表），训练/验证/测试划分沿用8:1:1分层抽样。

## 模型保存

- 路径A（微调）的权重、分词器配置通过`save_pretrained()`保存至`models/essay-scorer-finetuned/<version>/`；路径B（自定义构建）的权重通过`torch.save`保存至`models/essay-scorer-custom/<version>/`，两者分开存放，便于对比管理。
- 两条路径都保留训练日志（loss曲线、QWK随epoch变化曲线）一并存放，版本号按训练日期或迭代序号命名，便于答辩时展示训练过程而非只有最终数字。
- Agent侧的Prompt模板、LangGraph图结构配置以YAML/JSON形式保存在`src/agents/config/`，与模型权重区分管理。

## 模型预测

- 提供`EssayScorer`类，封装`load()`/`predict(text, essay_set) -> {"score": float, "traits": dict | None, "source": "finetuned"|"custom"|"ensemble"}`接口，内部可配置调用路径A、路径B，或对两者输出做加权融合，作为`01-系统架构与Agent设计.md`中`essay_scorer_tool`的底层实现。
- 预测时需要处理"用户提交的作文主题不在训练用的8个essay_set范围内"的情况（真实应用场景），参考`02-数据集与数据处理方案.md`"面向真实应用场景的适配"一节的通用性回退策略。

## 评估与答辩展示要点

- 训练完成后，在测试集上分别报告路径A、路径B的QWK分数，并与ASAP-AES竞赛公开的baseline/论文结果做量级对比（不追求超过SOTA，但要能说明"我们的模型达到了合理、可信的水平"）。
- 保留一版"路径A vs 路径B"的横向对比（QWK、训练耗时、参数量），以及路径A"微调前(零样本) vs 微调后"的效果对比，这是证明"两条模型路径确实由团队自己完成且有效"的直接证据，对应要求文档"注意"第2、3条的合规要求。
