# 07 数据处理与前端操作手册

> 负责人：林奕琳。负责范围：数据获取/清洗/EDA/预处理（`src/data_pipeline/`）+ Streamlit四页面（`app.py`）+ SQLite存储（`src/storage/db.py`，为历史进步仪表盘服务）。

## 一、负责范围里有哪些文件

| 文件 | 作用 |
|---|---|
| `src/data_pipeline/download.py` | 从HuggingFace镜像下载ASAP-AES原始数据，合并成`data/raw/asap_aes_merged.parquet` |
| `src/data_pipeline/clean.py` | 清洗：编码/空评分/异常长度/占位符/去重，产出`data/processed/essays_clean.csv` |
| `src/data_pipeline/eda.py` | 出EDA图表到`data/processed/eda/` |
| `src/data_pipeline/preprocess.py` | 分数归一化 + 8:1:1分层划分，产出`train/val/test.csv`和`score_ranges.json` |
| `app.py` | Streamlit入口，四个页面：提交批改/反馈详情/历史进步仪表盘/写作知识库问答 |
| `src/storage/db.py` | SQLite读写（`save_submission`/`get_user_history`），历史进步仪表盘依赖它 |

## 二、操作手册（照抄可复现）

### 1. 环境变量（数据下载/训练服务器上跑，本地开发机pip装不了大部分依赖）

```bash
export HF_ENDPOINT=https://hf-mirror.com   # 国内访问HuggingFace要走镜像
export HF_HUB_DISABLE_XET=1                # 否则大文件下载会卡死在0字节，见CLAUDE.md"不要重新踩的坑"
export HF_HOME=<你自己的缓存目录>           # 训练服务器上必须指到/data下，根分区是满的
```

### 2. 数据处理四步（必须按顺序跑）

```bash
python -m src.data_pipeline.download    # 下载并合并，12976条原始数据
python -m src.data_pipeline.clean       # 清洗：12976 -> 12879条（剔除97条空评分/过短/重复）
python -m src.data_pipeline.eda         # 出图表到 data/processed/eda/
python -m src.data_pipeline.preprocess  # 归一化 + 分层划分，同时生成 score_ranges.json
```

如果想换回Kaggle官方数据源（比如答辩被问起"为什么不用官方渠道"）：把`training_set_rel3.tsv`放到`data/raw/`下，`clean.py`会自动优先用它，不需要跑`download.py`。**当前团队决策是不追求这个**（HF镜像内容与官方一致，Kaggle官方需要账号手动接受比赛规则，见`Docs/Progress.md`第五轮记录），除非被明确要求，不用主动切换。

### 3. 跑Streamlit

```bash
streamlit run app.py
```

浏览器访问`http://localhost:8501`（本地开发）或`http://121.41.238.92:8503`（部署后，端口见毛陈荣的`08`号文档）。

### 4. 快速验证（不需要装完整依赖）

```bash
PYTHONPATH=. python scripts/smoke_test_nodes.py
```

`test_db_roundtrip`这部分单独验证了SQLite的读写往返，不依赖任何重量级包。

## 三、四个页面分别是什么状态（诚实现状，答辩别说错）

| 页面 | 状态 |
|---|---|
| 提交批改 | 真实：调用完整LangGraph（校验→检索→评分→语法检查→反馈→辅导） |
| 反馈详情 | 真实：展示量化评分（真实模型）、分项评分（启发式，非多头训练，见下）、定性反馈/修改建议（真实DeepSeek生成）、语法问题列表（真实规则库检测） |
| 历史进步仪表盘 | 真实：评分趋势折线图（SQLite真实历史数据）+ 分项雷达图（matplotlib极坐标图，Day3新增） |
| 写作知识库问答 | **仍是占位**：后端`retrieval_agent_node`是真实实现，但这个独立问答页面的前端交互没有接通，输入框是`disabled=True`。如果被问起，如实说"检索能力在提交批改流程里已经用上了，独立问答页面是Day1骨架、没来得及在Day3/4补" |

## 四、分项评分（trait_scores）到底是什么，别说成"多头模型"

雷达图三个轴（content/organization/language）**不是**单独训练出来的多头预测模型，是：
1. 三项的初始值 = 评分模型（微调DistilBERT占0.95、自建BiLSTM占0.05的非等权融合）输出的整体分，三项相同；
2. `grammar_check_node`（田溯开负责的模块）再用可观察的文本信号做启发式调整：language看语法错误密度，content看词汇丰富度（type-token ratio），organization看段落数/句长结构。

如果评委问"这是真正的多维度评分模型吗"，诚实回答：不是，是"整体分+启发式微调"，真正的多头训练需要额外的掩码多任务设计，属于诚实列出的遗留项（见`CLAUDE.md`"诚实的缺口"）。

## 五、EDA关键发现（答辩可以直接讲的点）

见`data/processed/eda/`下的图表，讲的时候可以带这几个数据支撑：
- 清洗后12879条样本，覆盖全部8个essay_set。
- 各essay_set样本量不均衡：essay_set 8只有约578条训练样本，明显少于其他集合（1252~1440条），这也是essay_set 8评分QWK偏低的根因之一（详细诊断见`09`号文档"essay_set 8误差诊断"）。
- 各essay_set分值范围差异很大（比如essay_set 3是0-3分，essay_set 8是15-60分），所以模型训练/评估都要按essay_set分别归一化、分别算QWK，不能混在一起看。

## 六、可能被追问的问题

- **"为什么不用Kaggle官方数据集？"** —— HF镜像内容与Kaggle官方完全一致，官方渠道需要账号并手动接受比赛协议（用团队的Token测试过，卡在这一步），非官方≠不可信，如果评委坚持可以现场展示切换方式（见上方"操作手册"）。
- **"数据清洗剔除了多少、为什么？"** —— 97条（评分缺失/异常长度/重复），比例约0.75%，不是大改动。
- **"分数归一化为什么要按essay_set分别做？"** —— 因为不同essay_set分值范围完全不同（0-3到0-60都有），不分别归一化的话模型会被分值范围大的题目集主导。
- **"历史进步仪表盘的雷达图三个维度可信吗？"** —— 如实说明见上方第四节，不要夸大成"多头模型"。
